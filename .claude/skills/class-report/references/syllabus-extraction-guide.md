# シラバス抽出ガイド

Step 0 でシラバスMarkdownからレポートスケジュールを抽出する際のルール。

## 抽出の基本方針

抽出は以下の優先順位で行う。上位で抽出できた場合、下位のロジックは使わない。

1. **優先パス**: `## 授業の進め方` の下位にある `### レポート・課題スケジュール` テーブル
2. **フォールバック**: `## 授業の進め方` 本文のパターンマッチ
3. **補完**: `## 授業計画` テーブルの各週の `授業内容` 列

抽出結果は **必ず Step 0 でユーザー確認を通す**。自動抽出は「人間による確認前提の下書き生成」と位置づける。

## 優先パス: 構造化テーブルからの読み取り

`class-syllabus` スキルが生成する最新シラバスには、以下の形式で構造化テーブルが出力される。

```markdown
## 授業の進め方

（本文）

### レポート・課題スケジュール
| レポート名 | 提出週 | テーマ |
|-----------|--------|-------|
| レポート1 | 第8週 | 字句解析の設計とプログラミング実践 |
| レポート2 | 第11週 | 構文解析の設計とプログラミング実践 |
| 最終レポート | 第14週 | 独自プログラミング言語の設計と実装 |
```

このテーブルが存在する場合、各行をそのまま読み取って抽出結果とする。以下の前提で読む:

- 列は `レポート名` / `提出週` / `テーマ` の固定3列
- `レポート名` は既に正規化済み（アラビア数字または特殊名）
- `提出週` は `第[N]週` 形式
- 合計行や注釈行がテーブル内にある場合は除外する

## フォールバック: フリーテキストからのパターンマッチ

構造化テーブルが無い既存シラバスに対しては、`## 授業の進め方` 本文のパターンマッチで抽出する。

### 積極パターン（採用する）

以下のいずれかにマッチした行を「レポート提出の言及」として採用する:

- `・レポート[丸数字または数字][:：][テーマ]（第[N]週提出）`
- `・[特殊名]レポート[:：][テーマ]（第[N]週提出）`  — 特殊名: `中間`, `期末`, `最終`
- `・レポート[N]は第[N]週に提出`
- `レポート[丸数字または数字]（第[N]週）`

**実例**:

```
・レポート①: 字句解析の設計とプログラミング実践（第8週提出）
・最終レポート: 独自プログラミング言語の設計と実装（第14週提出）
```

### 否定パターン（採用しない）

以下のような文脈は提出週として拾わない:

- 「レポート返却」「レポート講評」「レポート解説」を含む行
- 「期限内に提出すること」「遅延提出は減点」など一般的な提出ルールの言及
- 「レポートにフィードバックを返す」などのメタ記述

### 授業計画からの補完

`## 授業計画` テーブルの `授業内容` 列に「レポート[N]作成」「レポート[N]提出」などの記述がある場合、フォールバックで取りこぼしたレポートや提出週の補完情報として使う。ただし `## 授業の進め方` で抽出した情報を優先する。

**実例**:

```markdown
| 8 | 字句解析プログラムの実装・レポート①作成 | ... |
| 14 | 独自プログラミング言語の設計と実装（最終レポート作成） | ... |
```

## レポート名の正規化ルール

どの抽出パスでも、最終的にレポート名は以下の形式に統一する。

| シラバス表記 | 正規化後 |
|-------------|---------|
| `レポート①`, `レポート1`, `第1レポート` | `レポート1` |
| `レポート②`, `レポート2`, `第2レポート` | `レポート2` |
| `レポート③`, `レポート3`, `第3レポート` | `レポート3` |
| `課題A` | `レポート1`（英字→数字、出現順） |
| `課題B` | `レポート2` |
| `中間レポート` | `中間レポート` |
| `期末レポート` | `期末レポート` |
| `最終レポート` | `最終レポート` |

この正規化した名前を、ディレクトリ名・ファイル名・プレビュー表示で一貫して使用する。

## 抽出結果のユーザー確認

抽出後、Step 0 で以下のように確認を取る:

```
=== レポートスケジュール（抽出結果）===
| レポート名 | 提出週 | テーマ |
|-----------|--------|-------|
| レポート1 | 第8週 | 字句解析の設計とプログラミング実践 |
| レポート2 | 第11週 | 構文解析の設計とプログラミング実践 |
| レポート3 | 第13週 | 意味解析の設計とプログラミング実践 |
| 最終レポート | 第14週 | 独自プログラミング言語の設計と実装 |
```

`AskUserQuestion` で確認する:
- options: 「この内容で進む」「修正したい」「レポートを追加したい」「レポートを削除したい」

「修正したい」「追加・削除したい」が選択された場合、プレーンテキストで具体的な指示を聞き、抽出結果を更新する。

## 抽出が失敗した場合

以下のいずれかに該当する場合は、自動抽出を諦めて手動入力を求める:

- 構造化テーブル・フォールバックの両方でレポートの記載を検出できない
- 評価割合にはレポートの配点があるが、レポートの具体的な記述が一切ない

プレーンテキストで質問する:

```
レポート情報を自動抽出できませんでした。以下の情報を教えてください:
- レポートの名前（例: レポート1）:
- 提出予定週（例: 第8週）:
- テーマ（決まっている場合）:
```

複数レポートがある場合は繰り返し入力を受け付ける。
