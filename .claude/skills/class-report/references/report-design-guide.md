# レポート課題設計ガイド

Step 3（テーマ・課題内容の確定）と Step 4（課題詳細設計）で参照する設計ルール。

## Step 3: 統合フロー

シラバスからレポート情報を抽出済みの状態で、以下の手順で課題内容を確定する。シラバスの記載密度にかかわらず同じフローで処理する（不足部分は LLM が補完する）。

### 手順

1. **シラバス関連情報の整理**

   対象レポートについて以下をシラバスから整理・表示する:

   ```
   === シラバス記載の情報 ===
   レポート名: [名前]
   提出週: [第N週 または "未指定"]
   テーマ: [記載あり または "未指定"]
   概要: [記載あり または "未指定"]
   対応する到達目標: [シラバスの到達目標のうち関連する項目]
   対応する授業計画: [該当週の授業内容]
   ```

2. **不足部分を補完した課題案を1つ生成**

   以下のルールで補完する:

   - **テーマが未指定の場合**: 到達目標・授業計画から最も妥当な1案を生成する
   - **テーマのみ指定の場合**: テーマに沿った取り組み内容・提出物を補完する
   - **すべて指定済みの場合**: シラバス記載をそのまま構造化する

   提示フォーマット:

   ```
   === 課題案 ===
   テーマ: [テーマ名]
   課題の方向性: [1〜2文]
   想定する提出物: [レポート / プログラム / 複合 等]
   対応する到達目標: [対応項目の引用]
   対応する授業計画: [対応週の引用]
   ```

3. **ユーザー確認**

   `AskUserQuestion` で確認する:
   - options: 「この案で進む」「テーマを変更したい」「内容を修正したい」「別のテーマ候補を提案してほしい」

   「別のテーマ候補を提案してほしい」が選択された場合のみ、テーマ候補を3〜5個生成して提示する（生成ルールは本ガイド末尾の「テーマ候補生成ルール」を参照）。

   「テーマを変更したい」「内容を修正したい」が選択された場合、プレーンテキストで具体的な指示を聞いて反映する。

### 補完時の方針

- **シラバス記載を尊重**: シラバスに記載された情報は変更せず、不足部分のみを補う
- **到達目標との対応を明示**: 補完した内容は必ずシラバスの到達目標のいずれかに紐づける
- **具体性**: 抽象的な記述ではなく、学生が何に取り組むかが具体的に分かるテーマ・内容にする

## テーマ候補生成ルール

Step 3 でユーザーが「別のテーマ候補を提案してほしい」を選択した場合、以下の方針で3〜5個のテーマ候補を生成する。

### 生成の方針

1. **到達目標ベース**: 各到達目標に対応するレポートテーマを1つ以上考える
2. **授業計画ベース**: 授業計画の単元区切り（3〜5週のまとまり）ごとにテーマを考える
3. **実践重視**: 知識の再現ではなく、応用・分析・創造を求めるテーマにする
4. **多様性**: 到達目標・授業計画に応じて異なるアプローチ（調査／実装／分析／設計など）を含める

### 科目分野別のテーマ例

- プログラミング系: `theme-examples/programming-examples.md`
- アルゴリズム系: `theme-examples/algorithm-examples.md`
- コンパイラ系: `theme-examples/compiler-examples.md`

### 提示フォーマット

各テーマ候補には以下を含める:
- テーマ名（簡潔で具体的な名前）
- 1〜2行の説明（何をするか・何を学ぶか）

## 課題詳細設計ルール（Step 4）

### 必須項目

1. **課題の目的**（1〜2文）
   - 到達目標のどれに対応するかを明示する
   - 「〜を理解するために」「〜の能力を養うために」等の目的表現を使う

2. **課題の要件**
   - 必須要件: 最低限満たすべき条件（提出物・分量・形式）
   - 推奨要件: より高い評価を得るための条件（参考文献の活用、図表の作成等）
   - 分量の目安: レポートは2000〜4000字程度、プログラムは機能要件で指定

3. **取り組みの手順**
   - 3〜5ステップの段階的なガイダンス
   - 各ステップの目安時間は記載しない（学生によって異なるため）

4. **提出物の仕様**
   - ファイル形式（Markdown / PDF / ソースコード等）
   - 命名規則（例: `学籍番号_レポート1.md`）
   - 提出先（Moodle等）

5. **評価の観点（概要）**
   - ルーブリックの評価観点名を列挙する
   - 各観点の配点を記載する

### チームレポート固有の項目

6. **チーム構成に関する要件**
   - 役割分担もしくは貢献内容を記載するよう求める
   - チームメンバー全員の氏名・学籍番号をレポート冒頭に記載させる
   - チーム内でのレビュー・相互確認の推奨

役割分担の形態（明示的な担当分け / 共同作成）は課題の性質に応じて Step 4 時点で判断する。ユーザーが修正ループで調整したい場合は自由入力で指示を受け付ける。

## 提出物の分量目安

| 提出形式 | 分量の目安 |
|---------|----------|
| Markdown / テキストレポート | 2000〜4000字（図表除く） |
| PDF レポート | A4 3〜6ページ |
| プログラム（ソースコード） | 機能要件で指定（行数は指定しない） |
| プログラム + レポート | プログラム + 説明文1000〜2000字 |
