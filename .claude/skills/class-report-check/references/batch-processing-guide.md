# バッチ処理ガイド

Step 3 の提出物スキャン・対応表推定、および Step 5 の一括処理で参照する。

## 提出物の対応表推定

Moodle のエクスポート設定・教員の整理・提出形式によってディレクトリ構造は大きくブレる。構造の自動判定ではなく、**抽出された学生-提出物対応表をユーザーに確認**してもらうことで信頼性を確保する。

### 推定の手順

1. `submissions/` 配下を **再帰的に走査** して全ファイルを列挙する
2. 各ファイルについて、以下の情報源から学籍番号・氏名の候補を抽出する:
   - **パスに含まれる学籍番号**: 正規表現 `\d{7}`（7桁数字、標準）でマッチ
   - **フォルダ名・ファイル名に含まれる氏名**: 履修者名簿が提供されている場合は名簿の氏名と部分一致で照合
   - **Moodle形式のサフィックス**: `_assignsubmission_file_` などのサフィックスを除去した上で識別子を抽出
3. 各ファイルを学生単位にグルーピングする。同一学生の複数ファイルはまとめて1行にする

### 学籍番号のパターンカスタマイズ

デフォルトは7桁数字（`\d{7}`）。学校・年度によってケタ数や形式が異なる場合、Step 3 で `AskUserQuestion` の修正フローから正規表現を上書きできる。

### 対応表の表示

```
=== 提出物対応表（推定）===
| # | 学籍番号 | 氏名 | 検出されたファイル | 検出元 |
|---|---------|------|-------------------|-------|
| 1 | 2450001 | 山田太郎 | report.md, main.py | フォルダ名 |
| 2 | 2450002 | 佐藤花子 | 佐藤花子_report.pdf | ファイル名 |
| ? | ?       | ?     | misc.txt          | 識別不可 |
```

「識別不可」の行がある場合は、採点前に解消する必要がある。`AskUserQuestion` で以下を確認:
- options: 「この対応表で採点を開始する」「学生識別子（正規表現）を調整したい」「対応表を手動で修正したい」

「識別不可」の行が残っている状態では採点開始を許可しない。

### チーム提出の場合

チーム提出時は、チーム単位に対応表をグルーピングする。Step 3 で収集したチーム構成情報と照合してメンバー氏名・学籍番号を補完する。

#### 照合の優先順序

各提出フォルダ・ファイルについて、以下の順序でチームを特定する:

1. **チーム名で直接照合**: フォルダ名・ファイル名にチーム構成のチーム名（`チームA` `班1` `Team-Alpha` 等）が含まれていれば、そのチームに紐付ける。`specs/roster-team-csv.md` のチーム名揺らぎ対応に従い部分一致で判定する
2. **学籍番号からの逆引き**: チーム名が見つからない場合、学籍番号正規表現（デフォルト `\d{7}`）でフォルダ名・ファイル名から学籍番号を抽出し、チーム構成CSVを逆引きしてその学籍番号が属するチームに紐付ける（Moodle のチーム提出機能では代表者1名の学籍番号しか含まれないことが多いため、このパターンが頻出）
3. **両方失敗 → 識別不可**: 個人提出と同じく「識別不可」として表示し、ユーザー解消フローに乗せる。識別不可が残った状態では採点開始を許可しない

```
=== 提出物対応表（推定・チーム提出）===
| # | チーム名 | メンバー | 検出されたファイル | 検出元 |
|---|---------|---------|-------------------|-------|
| 1 | チームA | 山田太郎, 佐藤花子, 鈴木一郎 | report.md, src/main.py | チーム名（フォルダ名） |
| 2 | チームB | 田中次郎, 高橋三郎 | report.pdf | 学籍番号逆引き（代表者2450004） |
| ? | ?       | ?       | misc.zip          | 識別不可 |
```

## 対応ファイル形式

| 形式 | 拡張子 | 読み込み方法 |
|------|--------|-------------|
| Markdown | .md, .txt | Read ツール |
| Python | .py | Read ツール |
| JavaScript | .js, .ts | Read ツール |
| C/C++ | .c, .cpp, .h | Read ツール |
| Java | .java | Read ツール |
| Ruby | .rb | Read ツール |
| PDF | .pdf | Read ツール（後述のページ数別ルール参照） |

## PDF のページ数別の読み込みルール

Read ツールは1リクエストあたり最大20ページの制約がある。以下のルールで処理する:

| ページ数 | 処理 |
|---------|------|
| 20ページ以下 | 1回で全読み込み |
| 21〜100ページ | `pages` パラメータで20ページずつ分割読み込み（最大5回）、全文を統合して採点 |
| 101ページ超 | 採点保留。備考に「PDF[N]ページ（101ページ超）のため自動採点対象外・手動採点要」と記録し、成績CSVの点数列は空欄 |

## スキップ・保留するファイル

### 無視するファイル（採点対象外・記録不要）

- システムファイル: `.DS_Store`, `Thumbs.db`
- キャッシュ・管理ディレクトリ: `__pycache__/`, `.git/`, `node_modules/`
- アーカイブファイル: `.zip`, `.tar.gz`
- 実行ファイル: `.exe`, `.out`

### 採点保留（成績CSVに記録・点数は空欄）

以下に該当する場合、学生は「提出済み」と認識するが自動採点は行わず、成績CSVの備考列に理由を記録する:

- **画像単体提出**: `.png`, `.jpg`, `.jpeg`, `.gif`, `.svg` 等しか提出されていない
  - 備考: `画像単体のため採点保留・手動採点要`
- **非対応形式のみ**: 対応形式リストに無いファイルしか提出されていない（例: `.docx`, `.pptx` のみ）
  - 備考: `非対応形式（[拡張子]）のため採点保留・手動採点要`
- **PDF 101ページ超**: 上記 PDF ルール参照
  - 備考: `PDF[N]ページ（101ページ超）のため自動採点対象外・手動採点要`
- **読み込みエラー**: ファイルが破損している等で読み込みに失敗
  - 備考: `読み込みエラー・手動採点要`

採点保留の提出物は **サマリー統計（平均点・分布）から除外** する。未提出とは区別する。

画像は「単体のみ」のときだけ保留対象。レポートやコード内に埋め込まれた画像は採点対象とする提出物の文脈として考慮する。

## 1提出者が複数ファイルを持つ場合

すべての対応ファイルを読み込み、以下の優先順で主ファイルを判定する:

1. **レポートファイル**: `.md` > `.pdf` > `.txt`
2. **コードファイル**: メインファイル（`main.py`, `index.js` 等）> その他のコードファイル

採点では全ファイルの内容を総合的に評価する。

## 空ファイルの扱い

空ファイル（0バイトまたは内容のみ空白）は「空ファイル」として記録し、「未到達」で採点する。未提出や採点保留とは区別する。
