# レポート採点スキル

学生のレポート・成果物の提出物をルーブリックに基づいて自動採点し、観点別評価・フィードバックコメント・成績一覧CSVを出力するスキル。

## なぜこのスキルが必要か

レポートの採点は教員にとって最も時間のかかる業務の一つである。ルーブリックが明確に定義されている場合、採点基準は構造化されており、提出物をルーブリックの各段階と照合する作業は自動化の余地が大きい。アンカー採点方式（少数のサンプルで基準を確立してから一括採点）により、一貫した品質の自動採点を実現する。

## 前提条件

- 課題の出題文とルーブリックが存在すること（`/class-report` スキルで事前に作成、または手動で提供）
- 学生の提出物がディレクトリにまとめられていること
- 対応する提出物形式: Markdown (.md)、ソースコード (.py, .js, .c, .java 等)、PDF (.pdf)
- 画像単体は採点対象外（レポートやコード内に画像が含まれる場合は文脈として考慮する）

## デフォルト設定

| 項目 | 設定値 |
|------|-------|
| 配点 | 100点満点 |
| ルーブリック段階 | 3段階（理想的/標準的/未到達） |
| 理想的の標準点 | 観点配点 × 90% |
| 標準的の標準点 | 観点配点 × 70% |
| 未到達の標準点 | 観点配点 × 30% |
| CSV文字コード | UTF-8 BOM付き（Excel対応） |

ユーザーが明示的に変更しない限り、上記のデフォルト値を適用する。

## 対話ルール

ユーザーへの質問は場面に応じて適切な方式を使い分ける:

- **自由入力が自然な場面**（ファイルパス、採点の修正指示、チームメンバー情報など）: プレーンテキストで質問する
- **選択肢が明確な場面**（提出形態、配点方式、出力形式など）: `AskUserQuestion` ツールを使う

不明点や曖昧な点がある場合は、推測で進めずにユーザーに確認すること。

## 処理フロー

```
Step 0: 採点セッションの初期設定（科目・レポート名・提出形態）
Step 1: 課題内容・ルーブリックの読み込み
Step 2: ルーブリックの確認・配点確定
Step 3: 提出物ディレクトリのスキャン・受理
Step 4: アンカー採点（3件サンプリングで基準確立）
Step 5: 一括採点（残り全件）
Step 6: 採点結果レビュー
Step 7: ファイル出力（フィードバックMD + 採点一覧MD + 成績CSV）
Step 8: 完了レポート
```

### Step 0: 採点セッションの初期設定

プレーンテキストで順に質問する:

```
採点する科目の開講年度を教えてください（例: 2026）:
```

```
教科名を教えてください（例: コンパイラ）:
```

```
レポート名を教えてください（例: レポート1）:
```

`AskUserQuestion` で提出形態を確認する:
- options: 「個人提出」「チーム提出（チーム共通点数）」

**チーム提出の場合**: プレーンテキストで追加情報を収集する:
```
チーム構成を教えてください。以下のいずれかの方法で入力できます:
- チームメンバーリストのファイルパス（CSV等）
- チーム名とメンバーの一覧を直接入力
```

出力先の確定: `output/[年度]_[教科名]/reports/[レポート名]/`

### Step 1: 課題内容・ルーブリックの読み込み

Step 0 で確定した出力先ディレクトリ（`output/[年度]_[教科名]/reports/[レポート名]/`）に課題ファイルとルーブリックファイルが存在するか自動検出する。見つかった場合はパスを提示して確認する。見つからない場合はプレーンテキストで質問する:
```
課題MarkdownファイルとルーブリックMarkdownファイルのパスを教えてください
（/class-report で生成した場合の例:
  課題: output/2026_コンパイラ/reports/レポート1/レポート1_課題.md
  ルーブリック: output/2026_コンパイラ/reports/レポート1/レポート1_ルーブリック.md）

課題ファイルのパス:
ルーブリックファイルのパス:
```

1つのファイルに両方含まれている場合や、ルーブリックが別途存在しない場合にも対応する。

読み込んだ内容を表示する。課題文の各セクションからの抽出マッピングは以下の通り:

| 表示項目 | 課題文の抽出元セクション |
|---------|----------------------|
| 科目名 | `## 基本情報` テーブルの「科目名」行 |
| レポート名 | `## 基本情報` テーブルの「レポート名」行 |
| 課題の概要 | `## 課題の目的` セクション |
| 提出物の要件 | `## 提出物の仕様` セクション |
| 評価観点 | `## 評価基準` テーブル |

```
=== 課題情報 ===
科目名: [科目名]
レポート名: [レポート名]
課題の概要: [概要]
提出物の要件: [要件]

=== ルーブリック ===
| 評価観点 | 理想的 | 標準的 | 未到達 | 配点 |
|----------|--------|--------|--------|------|
| ... | ... | ... | ... | ... |
| 合計 | — | — | — | 100点 |

=== 点数換算基準（ルーブリックファイルから読み取り）===
| 段階 | 点数範囲 | 標準点 |
|------|---------|--------|
| ... | ... | ... |
```

ルーブリックファイルに「点数換算基準」セクションが含まれている場合、その値を読み取ってデフォルト設定より優先して使用する。

課題ファイルもルーブリックもない場合は、`/class-report` スキルでの事前作成を案内するか、プレーンテキストで直接入力を受け付ける。

ルーブリックファイルの「採点メモ欄」セクションは手動採点用のテンプレートであり、自動採点では無視する。

### Step 2: ルーブリックの確認・配点確定

`AskUserQuestion` でルーブリックを確認する:
- options: 「このルーブリックで採点を進める」「ルーブリックを修正・補完したい」

**配点の確定**: ルーブリックに配点が含まれていない場合、`AskUserQuestion` で方式を確認する:
- options: 「全観点均等配点（自動計算）」「観点ごとに配点を設定する」

均等配点の場合は合計点（デフォルト100点）から各観点の配点を自動計算する。

**点数換算基準の確認**:
```
=== 点数換算基準 ===
| 段階 | 点数範囲 | 標準点 |
|------|---------|--------|
| 理想的 | 80〜100% | 90% |
| 標準的 | 60〜79% | 70% |
| 未到達 | 0〜59% | 30% |
```

`AskUserQuestion` で確認する:
- options: 「この基準で進める」「基準を変更したい」

### Step 3: 提出物ディレクトリのスキャン・受理

**履修者名簿（オプション）**: プレーンテキストで質問する:
```
履修者名簿のファイル（CSV等）があれば、パスを教えてください
（ない場合は「なし」と入力してください。名簿がない場合は提出物から検出された学生のみを採点対象とします）:
```

名簿が提供された場合は、名簿に含まれる学生のうち提出物が検出されなかった学生を「未提出」として記録する。名簿がない場合は、提出物として検出されたもののみを採点対象とし、未提出の判定は行わない。

出力先ディレクトリ内に `submissions/` フォルダが存在するか自動検出する。見つかった場合はそのパスを提示して確認する。見つからない場合はプレーンテキストで質問する:
```
学生の提出物が格納されているディレクトリのパスを教えてください
（/class-report で生成した場合の例: output/2026_コンパイラ/reports/レポート1/submissions/）:
```

`AskUserQuestion` でディレクトリ構造を確認する:
- options: 「学生ごとのサブディレクトリ（学籍番号/氏名フォルダ）」「フラット配置（ファイル名で学生を識別）」

`references/batch-processing-guide.md` のスキャンロジックに従い、提出物を検出する。

スキャン結果を表示する:
```
=== 提出物スキャン結果 ===
提出者数: [N]名（チームの場合: [N]チーム）

ファイル形式の内訳:
  - Markdown (.md): [N]件
  - コード (.py/.js/.c 等): [N]件
  - PDF (.pdf): [N]件
  - スキップ（非対応形式）: [N]件

提出者一覧:
| # | 学籍番号/チーム名 | 氏名/メンバー | ファイル |
|---|------------------|--------------|---------|
| 1 | [番号] | [氏名] | [ファイル一覧] |
| 2 | [番号] | [氏名] | [ファイル一覧] |
...

未提出（検出されなかった）: [N]名
```

`AskUserQuestion` で確認する:
- options: 「この一覧で採点を開始する」「提出者リストを修正したい」「ディレクトリを変更したい」

### Step 4: アンカー採点（基準確立）

採点の一貫性を確保するため、全件採点の前にサンプル3件で基準を確立する。この方式により「同じ品質のレポートには同じ点数」を実現する。

`references/anchor-review-guide.md` に従い、3件をサンプリングする:
1. **記述量が多い1件**（高品質の可能性が高い）
2. **記述量が少ない1件**（低品質の可能性が高い）
3. **中程度の1件**

各サンプルを `references/scoring-guide.md` に従って採点し、結果を表示する:

```
=== アンカー採点: [学籍番号] [氏名] ===

### [観点1: 内容の正確さ・深さ]
評価: 標準的
点数: [N]点 / [M]点
根拠: ルーブリックの「標準的」の定義「〜〜〜」に合致。
      具体的には、提出物の〜〜〜の部分で〜〜〜を達成しているが、
      〜〜〜については不足している。

### [観点2: 構成・論理展開]
...

---
総合点: [N]点 / 100点
```

3件のアンカー採点後、`AskUserQuestion` で確認する:
- options: 「この採点基準で一括採点を開始する」「採点結果を修正したい」「別の3件でやり直したい」

「採点結果を修正したい」場合はプレーンテキストで「どの提出者のどの観点を修正しますか？」と質問し、修正を反映する。

アンカー採点が確定した時点で、以降の一括採点ではこの3件を基準として参照する。

### Step 5: 一括採点

Step 4 で確定したアンカーを基準に、残り全件を採点する。

各提出物の処理:
1. ファイルを読み込む（`references/batch-processing-guide.md` のファイルタイプ別読み込みルールに従う）
2. `references/scoring-guide.md` に従い、アンカーとの相対比較を含めて各観点を評価する
3. `references/feedback-guide.md` に従い、フィードバックコメントを生成する

採点中の進捗をユーザーに報告する:
```
採点進捗: [N]/[M] 完了 — 現在: [学籍番号] [氏名]
```

**一貫性チェック（採点完了後に実施）**:
- 点数分布に極端な偏り（全員満点、全員最低点等）がないか確認する
- アンカーと同じ段階の評価を受けた提出物の根拠が一貫しているか確認する
- 問題があれば修正してから Step 6 に進む

### Step 6: 採点結果レビュー

全件の結果をサマリー表示する:

```
=== 採点完了サマリー ===
提出者数: [N]名 / 採点完了: [N]名 / 未提出: [N]名

点数分布:
  90〜100点: [N]名 ████████
  80〜 89点: [N]名 ██████
  70〜 79点: [N]名 ████
  60〜 69点: [N]名 ██
   0〜 59点: [N]名 █

平均点: [N]点 / 最高点: [N]点 / 最低点: [N]点 / 中央値: [N]点

観点別 評価分布:
| 評価観点 | 理想的 | 標準的 | 未到達 | 平均点 |
|----------|--------|--------|--------|--------|
| [観点1] | [N]名 | [N]名 | [N]名 | [N]点 |
| [観点2] | [N]名 | [N]名 | [N]名 | [N]点 |
| ... | ... | ... | ... | ... |
```

`AskUserQuestion` で確認する:
- options: 「この結果でファイル出力に進む」「特定の学生の採点を修正したい」「全体の採点傾向を調整したい」

「特定の学生の採点を修正したい」: プレーンテキストで学籍番号と修正内容を聞く。
「全体の採点傾向を調整したい」: プレーンテキストで調整内容を聞き（例: 全体+5点シフト）、`references/scoring-guide.md` の調整ルールに従い反映する。

修正後は再度サマリーを表示し、確定するまで繰り返す。

### Step 7: ファイル出力

`AskUserQuestion` で出力形式を確認する:
- options: 「3種類すべて出力（推奨）」「個別フィードバックのみ」「成績一覧CSVのみ」

出力先:
```bash
mkdir -p "output/[年度]_[教科名]/reports/[レポート名]/feedback"
```

出力するファイル:

**1. 個別フィードバックMarkdown**（提出者ごと）
`output/.../reports/[レポート名]/feedback/[学籍番号]_[氏名]_feedback.md`

`references/output-format.md` のフォーマットに従う。内容:
- 観点別評価（段階 + 点数 + 採点根拠）
- 総合点
- フィードバックコメント（良い点・改善点・次のアクション）

**2. 採点一覧Markdown**（教員確認用）
`output/.../reports/[レポート名]/[レポート名]_採点一覧.md`

全提出者の採点結果を表形式で一覧表示。

**3. 成績一覧CSV**（成績処理用）
`output/.../reports/[レポート名]/[レポート名]_成績.csv`

`references/csv-format.md` のフォーマットに従う。UTF-8 BOM付き。チーム提出の場合はメンバー全員に同一点数を展開する。

### Step 8: 完了レポート

```
=== レポート採点完了 ===
科目名: [科目名]
レポート名: [レポート名]
採点日: [YYYY-MM-DD]

採点結果:
  採点済み: [N]名 / 未提出: [N]名
  平均点: [N]点 / 最高点: [N]点 / 最低点: [N]点

出力先: output/[YYYY]_[教科名]/reports/[レポート名]/

生成ファイル:
- 個別フィードバック: feedback/ 配下 × [N]件
- 採点一覧: [レポート名]_採点一覧.md
- 成績CSV: [レポート名]_成績.csv
```

## 採点品質の担保

このスキルは自動採点としてそのまま使用されることを前提としている。品質を担保するため、以下の仕組みを組み込んでいる:

1. **アンカー採点方式**: 3件のサンプルで基準を確立し、ユーザーが承認してから一括採点に進む。これにより「教員が納得した基準」で全件を採点する
2. **ルーブリック厳密準拠**: 各観点の評価は必ずルーブリックの定義を引用し、どの段階に合致するかを明示する
3. **採点根拠の明示**: 点数だけでなく、なぜその評価になったかの根拠を全件記録する
4. **一貫性チェック**: 全件採点後に分布チェック・アンカー整合チェックを自動実施する
5. **教員レビューステップ**: Step 6 で全体サマリーを提示し、修正の機会を設ける

## 対応する提出物形式

| 形式 | 拡張子 | 読み込み方法 |
|------|--------|-------------|
| Markdown | .md, .txt | Read ツールで直接読み込み |
| ソースコード | .py, .js, .c, .java, .rb 等 | Read ツールで直接読み込み |
| PDF | .pdf | Read ツールで読み込み（20ページ以内） |

画像ファイル（.png, .jpg等）が単体で提出された場合はスキップし、採点一覧に「画像単体のため採点対象外」と記録する。レポートやコード内に埋め込まれた画像は文脈として考慮する。
