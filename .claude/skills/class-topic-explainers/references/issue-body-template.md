# Issue 本文テンプレート

トピック解説 Issue の本文は以下の5セクション構成で固定する。順序・見出しレベルは変えない。学生がどの Issue を見ても同じ構造で情報を探せるようにすることで、資料としての信頼性を保つ。

## タイトル規約

```
[YYYY 教科名] 第N回 節名
```

- `YYYY`: 開講年度（例: `2026`）
- `教科名`: シラバスの科目名（略称ではなく正式名称）
- `第N回`: 授業回番号。1桁でも 2 桁でも `第1回` / `第10回` と自然に表記（`第01回` のようなゼロ埋めは不要）
- `節名`: Step 2 で抽出した節名そのまま

**例:**
- `[2026 ソフトウェア工学特論] 第1回 授業概要と3本柱`
- `[2026 コンパイラ] 第5回 DFA の実装`

教科名や節名に角括弧 `[]` を含めてはならない（タイトルパーサが誤動作する）。

## 本文フォーマット

```markdown
## 概要

（節本文を要約した 2〜3 文、合計 150 字以内。節の主題を端的に伝える）

## この節で学ぶこと

- キーコンセプト1
- キーコンセプト2
- キーコンセプト3
（節本文のスライド見出しをすべて列挙）

## 解説

（節本文の詳述。箇条書き・表・mermaid を活用して本文1箇所にまとめる）

### （小見出し）

...

## 関連情報

- **到達目標**: ルーブリック項目 [番号]（例: R1, R2）
- **MCC 対応**: V-D-*-*（該当するコード）
- **提出物**: （該当回のみ）レポートN / テストN
- **スライド**: [classNN.md の該当節](https://github.com/owner/name/blob/main/output/YYYY_[教科]/slides/classNN.md#節名アンカー)

## 議論・質問

質問や気づきがあればこの Issue へコメントしてください。
```

## 各セクションの書き方

### ## 概要

- 長さ: **150 字以内、2〜3 文**
- 目的: 学生が Issue を開いた瞬間に「この節で何を扱うか」を把握できるようにする
- 生成方法: 節本文の全スライドの内容から LLM で要約
- 要約が困難な場合（節本文が極端に短い、内容が断片的）のフォールバック: キーコンセプトを1文に並べる（例: 「この節では〇〇、△△、□□について扱います」）

### ## この節で学ぶこと

- キーコンセプト = 節本文のスライドの最上位見出し（章区切り以降、次の章区切り直前までの各スライドから抽出）
- 表紙・目次スライドの見出しは含めない
- 箇条書きで全件列挙（3〜7 個程度）
- Issue タイトルや概要セクションとの重複は気にしない（箇条書きは目次的に機能するため冗長でも OK）

### ## 解説

最も情報量の多いセクション。学生が講義後に復習する際、スライドだけでは補えない詳細をここで展開する。

**書き方の指針:**

- 節本文の内容を忠実に反映する。スライドに無い情報を推測で追加しない
- スライドで箇条書きになっていた部分は箇条書きのまま、補足説明があれば文章で添える
- 図表は以下の使い分け（`mermaid-usage-guide.md` 参照）:
  - フロー・時系列・状態・構造・階層 → mermaid
  - 比較 → Markdown 表
  - 数値・割合 → テキスト
- 例題や用例はスライドのものをそのまま転記しつつ、解法や補足をテキストで追加してよい
- 小見出し（`###`）で論点を分けてよい。長文を1つの見出し下に詰め込むより、学生が目的の情報を探しやすくなる

**してはいけないこと:**

- 複数の独立したトピックを「コメント分割」で書き分ける（本文1箇所にまとめる方針）
- スライドに無い具体的な数値・固有名詞・引用を追加する（正確性が担保できないため）

### ## 関連情報

箇条書きで以下を並べる。情報がない項目は省略してよい（例: 提出物該当回でなければ「提出物」行を省く）:

- **到達目標**: 当該回に紐付くルーブリック項目番号（シラバスから取得）
- **MCC 対応**: V-D-* の対応コード（シラバスから取得）
- **提出物**: 該当回のみ、レポート名やテスト名を書く
- **スライド**: スライド MD の該当節へのリンク（次節参照）

### ## 議論・質問

固定の案内文を入れる:

```
質問や気づきがあればこの Issue へコメントしてください。
```

学生がコメントで質問できる旨を明示する。Issue を「書き込み可能な場」として認識させるのが目的。

## スライド逆リンクの組み立て

### URL 形式

```
https://github.com/[owner]/[name]/blob/main/output/[YYYY]_[教科]/slides/classNN.md#[節名アンカー]
```

- `owner` / `name`: Step 0.3 で `gh repo view --json nameWithOwner` から取得した値
- `main`: ブランチ固定（プロジェクトの Git ワークフローで main 運用を前提とする）
- `[YYYY]_[教科]`: シラバスから決まる教科ディレクトリ名
- `classNN.md`: 回番号を2桁ゼロ埋めしたファイル名（例: `class03.md`）
- `[節名アンカー]`: GitHub の slugger 規則で変換した文字列

### アンカー生成規則（GitHub slugger）

- 小文字化（日本語・全角文字はそのまま）
- スペースをハイフンに変換
- 記号（`.`, `,`, `(`, `)` 等）は基本的に除去
- 先頭・末尾のハイフンは削除
- 重複見出しは `-1`, `-2` が付く（本スキルでは1スライド内で同一見出しが出ないため通常は考慮不要）

**例:**

| 節名 | アンカー |
|------|---------|
| 授業概要と3本柱 | `授業概要と3本柱` |
| DFA の実装 | `dfa-の実装` |
| 評価方法と学修単位 | `評価方法と学修単位` |

節が1個の回（目次・章区切りなし）では、**表紙の `## 第N回 XXX` のサブタイトルがアンカー先**になる:

- 例: `## 第1回 ガイダンス・環境構築` → アンカー `第1回-ガイダンス環境構築`（中点は GitHub 仕様上そのまま残る場合もあり、実装側で実際のアンカーを検証して調整する）

アンカーが効かない場合でもファイル全体へのリンクとしては機能するため、致命的ではない。

## 具体例

実際に生成される Issue の例:

**タイトル:**
```
[2026 コンパイラ] 第3回 字句解析の基礎
```

**本文:**
```markdown
## 概要

字句解析の役割とトークンの概念を押さえ、DFA を用いた字句解析器の構成方法を学ぶ。lex ツールによる実装例も示す。

## この節で学ぶこと

- トークンとは何か
- DFA による字句解析
- lex ツールの使い方
- 実装時のよくある落とし穴

## 解説

### トークンとは

字句解析はソースコードを**トークン列**に変換する処理である。トークンは「意味を持つ最小単位」で、識別子・キーワード・演算子・リテラルなどが該当する。

### DFA による字句解析

正規表現で記述した字句パターンは、以下の流れで DFA（決定性有限オートマトン）に変換される:

```mermaid
flowchart LR
    A["正規表現"] --> B["NFA"]
    B --> C["DFA（部分集合構成）"]
    C --> D["字句解析器"]
```

DFA は各状態で「次の入力文字が何なら、どの状態に遷移するか」を一意に決める遷移表を持つ。この遷移表を辿ることで、線形時間でトークン切り出しが可能になる。

...

## 関連情報

- **到達目標**: ルーブリック項目 R2
- **MCC 対応**: V-D-5-2
- **スライド**: [class03.md の字句解析の基礎](https://github.com/atsuki-seo/NITYC-MCC-Tools/blob/main/output/2026_コンパイラ/slides/class03.md#字句解析の基礎)

## 議論・質問

質問や気づきがあればこの Issue へコメントしてください。
