# 授業トピック解説ドキュメント生成スキル

授業用 Marp スライドから節単位にトピック解説ドキュメント（GitHub Issue）を生成し、スライドからのリンクを張り戻すスキル。学生が各節の内容を深く参照できる補助資料として Issue を運用する。

## なぜこのスキルが必要か

Marp スライドは授業中の視覚資料として最適だが、紙幅の制約から補足情報や図解を盛り込みづらい。Issue は mermaid をネイティブレンダリングでき、コメントによる質疑応答の場にもなるため、スライドから切り出した節単位で「詳しく読む」先として機能する。スライドと Issue の双方向リンクが張られていれば、学生は講義時にスライドから深掘りに飛び、復習時は Issue 側から議論履歴ごと参照できる。

## 前提条件

- シラバス情報が `class-syllabus-parse` スキルで事前に読み込まれていること
- 対象教科の `output/YYYY_[教科]/slides/classNN.md` が存在すること（`class-slides` スキルで事前に生成）
- カレントディレクトリが Git リポジトリで、`gh` コマンドで認証済みかつ Issue 機能が有効なリポジトリであること

## デフォルト設定

| 項目 | デフォルト |
|------|-----------|
| Issue 作成先 | カレント Git リポジトリ（`gh repo view` で動的取得） |
| 分解粒度 | 節 = 1 Issue（`references/slide-section-parser.md` 参照） |
| Issue タイトル | `[YYYY 教科名] 第N回 節名` |
| Issue 本文構成 | 概要 / この節で学ぶこと / 解説 / 関連情報 / 議論・質問（`references/issue-body-template.md` 参照） |
| 図解方式 | フロー・時系列・状態・構造・階層に該当する説明は mermaid で積極図示。比較は表、数値はテキスト（`references/mermaid-usage-guide.md` 参照） |
| メタデータ | ラベル・Milestone・Assignee は設定しない（リポジトリ側の Actions/テンプレートで運用） |
| スライド逆リンク | 章区切りスライド直下に `**関連ドキュメント**: [この節の解説](URL)` を追記 |
| マッピング管理 | `output/YYYY_[教科]/issues-map.yml`（git 管理下） |

ユーザーが明示的に変更しない限り、上記を適用する。

## 対話ルール

ユーザーへの質問は場面に応じて適切な方式を使い分ける:

- **自由入力が自然な場面**（回番号、修正指示の内容など）: プレーンテキストで質問する
- **選択肢が明確な場面**（対象範囲、確定か修正か、節名変更時の扱いなど）: `AskUserQuestion` ツールを使う

不明点は推測せずユーザーに確認する。

---

## 処理フロー

```
Step 0: 前提条件チェック
Step 1: 対象範囲の確認（全回 / 特定回）
Step 2: スライド MD の解析と節抽出
Step 3: 全節の骨子生成
Step 4: 骨子プレビューと修正ループ
Step 5: 節ごとの詳細化ループ（生成→プレビュー→修正→投稿を対象節すべてに連続適用）
Step 6: スライド MD への逆リンク書き戻し
Step 7: 完了レポート
```

### Step 0: 前提条件チェック

以下を順に確認し、失敗した時点で中断してユーザーに対処を案内する。致命的前提が揃わないまま先に進んでも意味のある処理ができないため早期失敗させる。

#### Step 0.1: シラバス情報の読み込み

シラバス解析は `class-syllabus-parse` スキルに委譲する。会話コンテキストに既にシラバス情報が読み込まれているかを確認し、以下のいずれかで処理する:

- **既に `class-syllabus-parse` で読み込み済みの場合**: 抽出結果を再利用して Step 0.2 へ進む
- **読み込まれていない場合**: 事前実行を案内して中断する:
  ```
  トピック解説ドキュメントの生成にはシラバス情報の解析が必要です。
  先に `/class-syllabus-parse` スキルを実行してシラバスを読み込んでください。
  ```

シラバス Markdown がそもそも存在しない場合は `/class-syllabus` スキルでの事前作成を案内する。

`class-syllabus-parse` 実行後、コンテキストには `specs/syllabus-markdown-schema.md` § 9.1 の契約通り、基本情報・授業計画（全週分）・到達目標・ルーブリック・レポート/試験スケジュール・概要・MCC対応が残っている前提で以降のステップを進める。

#### Step 0.2: スライド MD の存在確認

シラバスから特定される教科ディレクトリ（`output/[開講年度]_[教科名]/`）配下の `slides/classNN.md` を確認する。

- 1つも存在しない場合: `/class-slides` スキルの事前実行を案内して中断する:
  ```
  このスキルは `class-slides` が生成したスライド MD を入力として使います。
  先に `/class-slides` を実行してスライドを作成してください。
  ```
- 一部のみ存在する場合: 存在する回だけが処理対象になる旨を伝え、Step 1 へ進む

#### Step 0.3: Git リポジトリと gh 認証の確認

以下を順に実行し、一つでも失敗したら中断する:

```bash
git rev-parse --show-toplevel                                    # リポジトリルート取得
gh auth status                                                   # 認証確認
gh repo view --json nameWithOwner,hasIssuesEnabled               # リポジトリ情報取得、Issue 機能確認
```

失敗時の案内例:

- git: 「カレントディレクトリが Git リポジトリではありません。`git init` 済みのディレクトリで実行してください」
- gh 認証: 「`gh auth login` で GitHub に認証してから再実行してください」
- Issue 機能無効: 「対象リポジトリで Issue 機能が無効になっています。リポジトリ設定で Issues を有効化してください」

取得した `owner/name` は Issue URL の組み立てとマッピングファイルに使うため保持する。

### Step 1: 対象範囲の確認

`AskUserQuestion` で範囲を確認する:

- options: 「シーズン全回一括」「特定回のみ」

「特定回のみ」が選択された場合、プレーンテキストで回番号を聞く（複数可: 例「1,3,5」「7-9」）。

指定回番号のスライド MD がない場合は、存在する有効な回番号リストを提示してやり直す。

### Step 2: スライド MD の解析と節抽出

対象範囲の各回について、スライド MD を読み込んで節を抽出する。詳細アルゴリズムは `references/slide-section-parser.md` に記載。概要は以下の通り:

1. スライドを `---` で分割
2. **目次スライド**（「# 目次」で始まり本文が箇条書き）の有無を判定
3. 目次あり: 目次項目と**章区切りスライド**（見出しのみで本文なし）を突合し、節名と本文範囲を確定
4. 目次なし: 節が1個とみなし、表紙の「## 第N回 XXX」を節名、表紙以外の全スライドを本文範囲とする
5. 目次はあるが章区切りと一致しない等のフォーマット異常: 当該回をスキップして警告（他の回の処理は続行）

抽出結果として各節について以下を保持する:

- 回番号 / 回タイトル / 節名
- 節本文に含まれるスライドの見出し一覧（キーコンセプト抽出に使う）
- 節本文の原文（Issue 本文生成の入力）
- 章区切りスライドの位置（Step 6 で逆リンクを書き戻す先）

### Step 3: 全節の骨子生成

対象範囲の各節について以下を作成する:

- 回番号 / 節名
- キーコンセプト（節本文のスライドの最上位見出し 3〜7 個）
- 到達目標（回単位、シラバスのルーブリック項目番号）
- MCC 対応（回単位、V-D-*）
- 提出物該当（該当回のみレポート名・テスト名、そうでなければ `—`）
- 既存 Issue 番号（`output/YYYY_[教科]/issues-map.yml` から、該当なしなら `—`）

**マッピングファイルの読み込み**: `output/YYYY_[教科]/issues-map.yml` が存在すればロードし、各節について `section_name` で突合する。

**節名変更の検出**: マッピングに存在するが新しい節リストに無い `section_name`（`deprecated: true` でないもの）と、新しい節リストに存在するがマッピングに無い `section_name` を突き合わせる。同じ回にそれぞれ1件ずつあれば「節名がリネームされた可能性」として Step 4 で確認する。複数件あれば位置的な推論を避け、すべて「新規 / 既存消失」として扱う（誤マッピングを防ぐ）。

### Step 4: 骨子プレビューと修正ループ

全節の骨子一覧を Markdown テーブルで提示する:

```
=== 骨子案 ===
| 回 | 節名 | キーコンセプト | 到達目標 | MCC | 提出物 | 既存 |
|----|------|--------------|---------|-----|-------|------|
| 1 | 授業概要と3本柱 | 3本柱, 評価方法, 進め方 | R1-R3 | V-D-*-* | — | — |
| 1 | 評価方法と学修単位 | 評価比率, 学修単位, 提出物 | R1-R3 | V-D-*-* | — | — |
| 3 | 字句解析の基礎 | トークン, DFA, lex | R2 | V-D-*-* | — | #50 |
| ... |
```

**節名変更候補がある場合**、該当行の「節名」列に `旧: 字句解析の基礎 → 新: 字句解析の導入` のように併記して、ユーザーに見せる。

#### Step 4.1: 節名変更の確認（該当があれば）

`AskUserQuestion` で各候補について確認する:

- options: 「リネーム（既存 Issue のタイトル・本文を更新）」「クローズして新規作成」「両方残す（旧 Issue は deprecated フラグ）」

「両方残す」を選ぶと旧マッピングに `deprecated: true` を付け、新しい節は新規 Issue として作成する。

#### Step 4.2: 骨子の承認

`AskUserQuestion` で確認する:

- options: 「この骨子で進む」「節を指定して修正したい」「全体を再生成したい」

「節を指定して修正したい」はプレーンテキストで修正指示を聞く（例「第3回の字句解析の節は今回スキップして」「第5回にキーコンセプト追加して」）。スキップ指示を受けた場合は該当節を処理対象から外す。確定するまで繰り返す。

### Step 5: 節ごとの詳細化ループ

対象節数分、以下を繰り返す（骨子順）。各節の Step 5.1〜5.4 を連続で実施し、1節完了したら確認なしで自動的に次の節へ進む。ユーザーから能動的に中断指示があった場合は即座に Step 6 へ移る。

#### Step 5.1: Issue 本文の生成

骨子に従って Issue 本文を生成する。詳細フォーマットは `references/issue-body-template.md` を参照。主要ルール:

- **概要**: 節本文（複数スライド）を要約して 2〜3 文、150 字以内。要約が困難な場合はキーコンセプトの箇条書きにフォールバック
- **この節で学ぶこと**: キーコンセプト（節本文のスライド見出し）を箇条書きで全件列挙
- **解説**: 節本文の内容を本文1箇所にまとめて詳述。箇条書き・表・mermaid を活用。mermaid の使用基準は `references/mermaid-usage-guide.md` に従う
- **関連情報**: 到達目標 / MCC / 提出物（該当回のみ）/ スライドへの逆リンク（blob URL + 節見出しアンカー）
- **議論・質問**: 学生が書き込むためのスペース。空欄（案内文のみ）

**スライドへの逆リンク URL**:

```
https://[owner]/[name]/blob/main/output/[YYYY]_[教科]/slides/classNN.md#[節見出しアンカー]
```

節見出しアンカーは GitHub の slugger 規則で生成する（小文字化・スペース→ハイフン・日本語はそのまま）。詳細は `references/slide-section-parser.md` § アンカー生成 を参照。

#### Step 5.2: プレビュー提示

生成した Issue の以下を表示する:

- Issue タイトル
- Issue 本文全体（セクション見出し込み）
- mermaid を含む場合はその旨を注記（GitHub 上でネイティブレンダリングされるため Viewer 案内は不要）

#### Step 5.3: 修正指示ループ

`AskUserQuestion` で確認する:

- options: 「この内容で投稿」「内容を修正したい」「再生成したい」

「内容を修正したい」はプレーンテキストで修正指示を聞き（例「mermaid の矢印方向を逆にして」「例を追加して」）、差分をハイライトで提示しつつ更新する。確定するまで繰り返す。

#### Step 5.4: GitHub Issue に投稿

既存 Issue の有無とユーザー選択（Step 4.1）に応じて、`gh` コマンドを使い分ける:

- **新規作成** (既存なし): `gh issue create --title "..." --body "..."`
- **更新** (既存あり、本文に差異あり): `gh issue edit [番号] --title "..." --body "..."`
- **既存のまま** (既存あり、本文が完全一致): API 呼び出しをスキップ
  - 既存本文は `gh issue view [番号] --json body` で取得して比較
- **リネーム**: `gh issue edit [番号] --title "..." --body "..."`（タイトル含めて更新）
- **クローズして新規作成**: `gh issue close [番号] --comment "節名変更により新 Issue に移行: #[新番号]"` → `gh issue create`
- **両方残す**: 旧 Issue はそのまま、新 Issue は `gh issue create`。旧マッピングに `deprecated: true` を付ける

投稿成功後、`issues-map.yml` を更新する:

- 該当回の `sections` 配列に `section_name` / `issue_number` / `issue_url` / `created_at` / `updated_at` を記録
- 既存エントリを更新する場合は `updated_at` のみ差し替え（`created_at` は維持）

個別失敗（認証切れ・ネットワーク断・API エラー等）は致命的ではないのでログに残して次の節へ続行する。Step 7 の完了レポートで列挙する。

### Step 6: スライド MD への逆リンク書き戻し

Step 5 で投稿に成功した全 Issue について、スライド MD にリンクを書き戻す。バッチで実行する（1節ごとに書き戻すと、途中失敗でスライド MD が中途半端な状態になるため）。

#### Step 6.1: 書き戻し位置

- **節が複数の回**: 各章区切りスライドの見出し行の直下（空行を挟んだ位置）に追記
- **節が1個の回**: 表紙スライド本文末尾（`---` の直前）に追記

#### Step 6.2: 追記テキスト

節が複数の場合:

```markdown
**関連ドキュメント**: [この節の解説](https://github.com/owner/name/issues/NN)
```

節が1個の場合は文言を少し変える（回 = 節なので「この回の解説」とする）:

```markdown
**関連ドキュメント**: [この回の解説](https://github.com/owner/name/issues/NN)
```

#### Step 6.3: 既存リンクの置換

再実行時に重複追記しないよう、`**関連ドキュメント**: \[この(節|回)の解説\]\([^)]+\)` にマッチする行を正規表現で検出し、存在すれば置換、なければ挿入する。

書き換えたスライド MD のパスは Step 7 の完了レポートに含める。

### Step 7: 完了レポート

**通常時**:

```
=== トピック解説ドキュメント生成完了 ===
科目名: [科目名]
担当教員: [教員名]
開講年度: [YYYY]
対象リポジトリ: owner/name

生成したトピック解説:
| 回 | 節名 | ドキュメントURL | 状態 |
|----|------|----------------|------|
| 1 | 授業概要と3本柱 | https://github.com/owner/name/issues/42 | 新規作成 |
| 1 | 評価方法と学修単位 | https://github.com/owner/name/issues/43 | 新規作成 |
| 3 | 字句解析の導入 | https://github.com/owner/name/issues/50 | リネーム |
| 5 | DFA の実装 | https://github.com/owner/name/issues/52 | 更新 |
| 7 | 構文解析入門 | https://github.com/owner/name/issues/54 | 既存のまま |
| ... |

スライドへの逆リンク書き戻し:
- output/YYYY_[教科]/slides/class01.md
- output/YYYY_[教科]/slides/class03.md
- ...

スキップした節:
- 第N回: スライドから節を抽出できませんでした（目次と章区切りが不一致）
- 第N回 [節名]: ユーザー判断でスキップ

マッピングファイル: output/YYYY_[教科]/issues-map.yml を更新しました
```

**個別失敗がある場合の追記**:

```
⚠ ドキュメント作成失敗（個別）
- 第N回 [節名]: gh issue create が失敗
  エラー: [gh のエラーメッセージ]
  対処: 認証状態（gh auth status）や Issue 作成権限を確認してください
```

状態列の取りうる値は「新規作成」「更新」「リネーム」「既存のまま」の 4 種類。Step 5.4 の判定に従う。

- シラバス Markdown 入出力契約: プロジェクトルートの `specs/syllabus-markdown-schema.md`
- Issue 本文のセクション構成とテンプレート: `references/issue-body-template.md`
- mermaid の使用基準と記法ガイド: `references/mermaid-usage-guide.md`
- スライド MD からの節抽出アルゴリズムとアンカー生成: `references/slide-section-parser.md`
- 既存ファイル検出・上書きポリシー: プロジェクトルートの `specs/file-output-policy.md`（本スキルは `issues-map.yml` と スライド MD への追記に適用）
